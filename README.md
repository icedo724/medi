## ⚠️ 데이터 보안 및 익명화 
본 프로젝트는 실제 기업 현장에서 수행된 과제이나, **계약에 따라** 다음과 같은 조치를 취했습니다.

1.  **데이터 익명화:** 모든 거래처정보와 기타 민감데이터 등은 가상의 데이터로 대체되었습니다.
2.  **스키마 표준화:** 원본 데이터베이스의 한글 테이블명(예: `[회사명]거래처현황`)은 영문 표기(예: `client_info`)로 변경하여 코드 가독성을 높였습니다.
3.  **민감 정보 제거:** API Key 및 내부 서버 접속 정보는 환경 변수(`.env`)로 분리하여 코드에 노출되지 않도록 처리했습니다. 

---

## 1. 프로젝트 개요
본 프로젝트는 내부 영업 데이터와 공공데이터를 결합하여 영업 효율을 높이고, 데이터 기반의 의사결정을 지원하기 위해 수행되었습니다.

기존의 직관에 의존한 영업 방식에서 벗어나, 객관적인 데이터를 기반으로 거래처 등급을 재산정하고 맞춤형 영업 전략을 수립하는 것이 목표입니다.

---

## 2. 프로젝트 목표
1. 내부 거래처 데이터 질 향상을 위한 데이터 수집 및 처리
2. 수집된 데이터를 통한 데이터베이스 구조 제안
3. 정리된 데이터를 통해 기존에 내부에서 사용하던 거래처 등급 재분류 및 요인 분석

---


## 3. 프로젝트 구조
```text
Health-Data-Analysis/
├── 01_Data_Collection/      # 공공데이터 포털 API 수집
├── 02_Data_Preprocessing/   # 데이터 정제 및 결합
├── 03_SQL_Warehouse/        # 분석용 데이터 마트 구축을 위한 SQL 쿼리
├── 04_Analysis_Modeling/    # RFM 고객 세분화 및 시각화
├── 05_Application/          # 통계 분석 GUI 도구
└── data/                    # (가상 데이터로 대체됨) 데이터 저장소
```

---

## 4. 핵심 분석 결과
등급 재분류: 기존에 내부에서 사용되던 주관에만 의지한 거래처 등급을 주관을 유지하면서 객관적인 데이터로 새로 정의함.

요인 분석: 높은 등급을 보유한 거래처들의 주요 특성과 요인을 실제 통계적 기법을 통해 검정

---

## 5. 기대 효과
내부 데이터 질적 향상: 기존 내부 데이터에 공공데이터를 연결하여 질적인 향상을 이뤄냈고 이후 거래처가 추가되거나 갱신되더라도 꾸준히 관리 될 수 있도록 그 기반을 마련하였습니다.

정교한 영업 타겟팅: 직관이 아닌 객관적 지표에 근거하여 '이탈 위험군'과 '잠재 우량 고객'을 식별함으로써 영업 리소스 낭비를 줄였습니다.

잠재 고객 발굴: 만약 새로운 거래처가 온다면 기존 거래처 데이터와 비교하여 향후 매출 방향성 예측 가능

---

## 6. 이슈
데이터 매칭 정확도 이슈: 

문제: 사내 데이터 상의 거래처명(예: 행복내과)과 공공데이터 상의 병원명(예: 행복내과의원)이 일치하지 않아 단순 조인(Join)이 불가능했습니다.

해결: 한글 유사도 알고리즘을 사용하는 RapidFuzz 라이브러리를 도입하여 텍스트 유사도를 계산함과 동시에 주소를 이용해 일정 범위 내의 후보군 중에서 직접 고르는 방식으로 해결하였습니다.



대용량 API 수집 이슈:

문제: 전국 병원 데이터를 한 번에 호출 시 API 연결 끊김 현상이 발생했습니다.

해결: 데이터를 페이지 단위(Paging)로 분할 수집하고, try-except 구문을 활용한 자동 재시도(Retry) 로직을 구현하여 데이터 수집의 안정성을 높였습니다.


UDI 코드 이슈:

문제: 장비에 대한 정보도 처리하여 내부 데이터와 완벽 연동하고자 하였지만 UDI코드가 제공되지 않는 장비들이 발견되었습니다.

해결: 장비의 분류명은 공통된 것을 확인하고 이에 대해서만 연동하였였습니다.
